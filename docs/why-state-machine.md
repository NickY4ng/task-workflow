# 为什么用状态机？

## 问题：Agent 产出总跑偏

AI 生成内容 → 人逐字检查 → 发现不对 → 反复修改 → 还是不对 → 再改...

**核心痛点：**
1. **理解偏差**：AI 以为自己懂了，其实没懂
2. **执行偏离**：做着做着就偏离了最初目标
3. **验证负担**：人成了"校对机器"，逐字核查
4. **上下文丢失**：改着改着忘了最初要做什么

---

## 方案：五步状态机

### 为什么分五步？

**每一步都是"检查点"**，防止偏差累积。

| 步骤 | 作用 | 不做的后果 |
|------|------|-----------|
| Step 1 复述确认 | 确保理解正确 | 理解错了，后面全错 |
| Step 2 方案设计 | 确认怎么做 | 做到一半发现方向不对 |
| Step 3 执行 | 按方案做 | 自由发挥，偏离目标 |
| Step 4 自检 | 交付前自验 | 带着错误交付 |
| Step 5 交付 | 确认完成 | 用户不满意，再改 |

### 为什么是"硬约束"？

**软约束 = 没约束**

如果 AI 可以自己判断"这一步差不多就行了"，那状态机就失效了。

硬约束：
- 必须用户说"确认"才能推进
- 必须用户说"改"才能回退
- AI 不自主决策

---

## 效果

**传统方式：**
```
用户：帮我写个方案
AI：写好了
用户：不对，要改A
AI：改了
用户：B也不对
AI：改了
用户：C还是不对...
（5-6轮后勉强可用）
```

**五步状态机：**
```
用户：帮我写个方案
AI：我的理解是...方向对不？
用户：对
AI：方案如下...确认吗？
用户：确认
AI：执行中...
AI：自检完成，交付
用户：OK
（2轮确认 + 1轮交付 = 一次性到位）
```

---

## 核心原则

**不是让 AI 更快，是让 AI 更准。**

速度 ≠ 质量
确认 → 质量 → 减少反复 → 真正的快
