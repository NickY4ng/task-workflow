# 为什么事件驱动？

## 问题：任务管理混乱

- 做着做着忘了当前在哪一步
- 多个任务混在一起，上下文串了
- 上次做到哪了？忘了

---

## 方案：事件驱动管理

### 每个任务 = 一个事件

```
事件ID: abc123
状态: Step 3/5 执行中
需求: 写数据平台方案
创建时间: 2026-05-06 14:00
更新时间: 2026-05-06 14:30
```

### 事件属性

| 属性 | 作用 |
|------|------|
| 事件ID | 唯一标识，不混淆 |
| 状态 | 当前在哪一步 |
| 需求 | 原始需求 + 确认后的需求 |
| 过程日志 | 每一步的操作记录 |
| 反馈 | 用户的修改意见 |
| 结果 | 输出文件 + 摘要 |

---

## 好处

### 1. 可追溯

任何时候都能回看：
- 最初需求是什么
- 中途改了什么
- 最终交付了什么

### 2. 可恢复

对话中断后：
- 读取事件状态
- 恢复到中断前
- 继续执行

### 3. 多任务并行

同时处理多个任务：
- 任务A: Step 3 执行中
- 任务B: Step 2 方案设计
- 任务C: Step 5 交付确认

切换上下文不丢失。

---

## 核心原则

**任务不是一次性的，是可管理的。**

像项目管理一样管理 AI 任务：
- 有ID
- 有状态
- 有记录
- 可追溯
